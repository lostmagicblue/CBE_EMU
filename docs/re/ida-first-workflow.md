# IDA 优先开发流程

## 目的

从这里开始，江湖 OL 服务端复刻统一采用下面这条默认循环：

`IDA/运行证据 -> 本地文档 -> 窄实现 -> 构建验证 -> 文档回写 -> 下一卡点`

不再先猜包、不再先堆 trace，也不再在没有结构体/字段/业务流程证据的情况下直接改 `src/mock-server.c` 或 `src/main.c`。

## 每次会话开始的强制顺序

1. 先读已有上下文：
   - 当前卡点对应的 `docs/re/*.md`。
   - `git diff` 中与该阶段相关的改动。
   - 现有运行现象：assert、正常控制台输出、可见界面状态。
2. 把本轮目标缩小成一个最小问题：
   - 一个 loading 卡点。
   - 一个 unhandled request。
   - 一个错误的场景进入位置。
   - 一个错误的 parser 分支。
3. 打开 IDA，按 `binary_name` 选择实例，不写死实例 id。
4. 先读通再动手：
   - 相关调用链。
   - 相关结构体或状态块。
   - 字段读写顺序。
   - 成功路径与失败路径。
   - 客户端下一步会发什么请求、切什么状态。
5. 在 `docs/re/` 新建或更新一份本地调查文档。
6. 只有调查文档达到“可实现”门槛后，才开始写代码。

## IDA 阅读最少完成项

开始实现前，至少要在文档里说明这些内容：

- 入口函数、上游调用者、下游关键调用。
- 涉及的 binary/module：`江湖OL.CBE` 还是某个 `mm*.cbm`。
- 关键状态属于哪个对象或全局块，偏移大致在哪里。
- 哪些字段只是被跳过，哪些字段会影响分支、场景、坐标、回调、UI。
- 成功路径里客户端会如何推进。
- 失败路径里客户端为什么卡住、反复加载或闪退。
- 仍未知的字段、偏移、对象类型，以及下一步如何确认。

如果这些信息还说不清，就继续读 IDA，不要急着实现。

## 推荐的 IDA MCP 使用顺序

1. `list_instances`
2. `analyze_function` 或小范围 `analyze_batch`
3. 确认目标函数后再 `decompile`
4. 只在必要时用 `py_eval` 做局部验证

约束：

- 先小范围分析，不做整包全量反编译。
- 文档里记录函数名、地址、binary 名称。
- 优先围绕请求解析、响应解析、场景切换、坐标写入、状态机推进这几类函数取证。

## 本地调查文档要求

调查文档统一放在 `docs/re/`。

命名规则：

- 同一阶段已有文档时，优先继续追加。
- 新阶段可新建 `YYYY-MM-DD-<phase>.md`。
- 如果是长期主题，例如 Penglai 场景切换，可以继续维护单一主题文档。

建议直接从 `docs/re/phase-investigation-template.md` 起手。

每份调查文档至少要包含：

- 当前卡点和最小目标。
- 运行时现象。
- IDA 目标函数与地址。
- 调用链 / 业务流程。
- 结构体或状态字段笔记。
- 请求/响应契约草图。
- negative evidence。
- 下一步准备实现什么，以及为什么只改这一小块。
- 验证清单。

## 实现前门槛

下面这些条件没满足时，不进入实现：

- 已锁定至少一个真实客户端函数和地址。
- 已说明客户端期待的响应、事件或状态推进点。
- 已区分“证据”与“猜测”。
- 已把结构体/字段/流程记录进 `docs/re/`。
- 本轮改动范围已经收敛到一个最小问题。

## 单轮推进循环

1. 选一个最小卡点。
2. 补 runtime 观察和 IDA 证据。
3. 更新本地调查文档。
4. 做一个窄改动。
5. `make`、运行、验证客户端是否自然推进。
6. 把结果、失败样例、未知字段回写文档。
7. 再决定下一轮卡点。

如果运行结果和文档假设不一致，优先回到 IDA 和文档，而不是连续追加试错响应。

## 与现有规范的关系

- 总体边界仍以 `docs/re/jianghu-ol-server-agent.md` 为准。
- 协议取证仍遵循 `.codex/skills/cbe-reverse-engineering/SKILL.md` 和 `.codex/skills/jianghu-protocol-forensics/SKILL.md`。
- 临时观测、trace、日志约束仍遵循 `docs/re/trace-policy.md` 与 `docs/re/logging-policy.md`。

这份流程文档的作用是把“先读通 IDA、先写本地文档、再实现”变成每次会话开始时的默认动作，而不是可选习惯。
