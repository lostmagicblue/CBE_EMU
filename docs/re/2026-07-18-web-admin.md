# 江湖 OL mock 服务网页后台

## 入口

服务端模式启动后，后台默认仅监听本机：

```text
http://127.0.0.1:19091/
```

后台访问密码固定为 `123456`。验证成功后服务端写入 `HttpOnly`、
`SameSite=Strict` 的进程内会话 Cookie；退出登录或重启服务后需要重新验证。

游戏协议仍使用 `127.0.0.1:19090`，两个监听端口由同一服务进程维护。

后台端口可通过以下任一种方式修改：

```text
--mock-admin-port=19091
CBE_MOCK_ADMIN_PORT=19091
```

管理端口固定绑定 `127.0.0.1`，不会跟随游戏服务的
`--mock-service-bind` 暴露到局域网。

## 首版功能

- 页面包含“账号管理”“游戏内容管理”“任务管理”“怪物管理”“商品管理”和“游戏内容更新管理”六个 Tab。
- 查看账号总数、账号在线/离线状态。
- 账号列表使用固定高度滚动区域，账号增加后不会无限拉长页面。
- 账号和场景列表使用 `sessionStorage` 分别保存滚动位置；点击列表项触发
  整页跳转后会恢复原来的 `scrollTop`，URL 锚点作为禁用脚本时的后备定位。
- 查看账号下的角色 ID、名称、等级、普通钱币和 W 币。
- 按角色增加普通钱币。服务端以铜为基础单位，页面同时换算金、银、铜。
- 按角色增加 W 币。
- 创建账号。
- 修改账号密码。
- 查看服务端 `web/fs/JHOnlineData` 中现有的 SCE 场景文件和文件大小。
- 选择 SCE 后直接从真实 SCE/MAP/GIF 资源生成完整 BMP 地图预览，并按场景
  像素坐标叠加 SCE 内置 NPC 与当前启用的服务端动态 NPC；不读取 `tmp`
  导出图片。空壳 SCE 仅在严格同名 MAP 存在时使用同名资源作为后备。
- 按场景查看、增加、编辑、停用或恢复服务端动态 NPC。
- 内置动态 NPC 使用 MySQL 覆盖项修改；删除覆盖项可恢复内置默认。自定义
  NPC 的覆盖项可直接删除。

普通钱币与 W 币操作复用服务端账户状态切换及角色数据库保存路径，写入
MySQL `account_roles`，不会维护第二份网页专用数据。

HTTP 请求解析、页面渲染和管理监听器位于 `src/web_admin_server.c`；
`src/mock-server.c` 只保留共享的账户/角色数据接口与服务循环接入点。
服务端不再轮询标准输入，也不再提供控制台管理命令。

动态 NPC 覆盖项保存在 MySQL `server_dynamic_npcs` 表。服务端启动时会执行
`CREATE TABLE IF NOT EXISTS`，完整建表语句也已加入 `server/mysql/schema.sql`
和 `server/mysql/migrate_add_dynamic_npcs.sql`。网页提交会校验：

- 场景必须是服务器目录中真实存在的 SCE 文件；
- Actor ID、坐标、类型和朝向必须在协议字段范围内；
- Actor 资源必须存在于服务端权威资源目录、可以正确解压解析，且引用的所有 GIF
  均存在并可解码；脚本由服务端解释，非空时对应服务端 XSE 必须存在；
- 显示名称和资源名满足客户端固定字段的 GBK 字节长度；
- 同一场景仍最多向客户端选择 4 个动态名称，超出时沿用任务优先策略。

Web 在 Actor 下拉框中展示权威资源目录里的完整 Actor 集合，不以
`bin/JHOnlineData` 的缓存状态判断资源是否存在。保存或恢复 NPC 时会实际解析
Actor 及其 GIF 依赖，将整组文件发布到具名资源目录后再写入启用状态；不会修改
或复制文件到任何客户端目录。客户端处理场景 NPC 时若打开 Actor/GIF 失败，
模拟器会以当前客户端身份发送 WT `18/7` 分块请求，校验累计 CRC，原子写入本地
`JHOnlineData` 后重试原文件操作。只有后台已发布的名称允许这条自动下载路径。
服务启动时会为升级前已经启用的动态 NPC 补发 Actor/GIF 发布项；历史数据库记录
若引用的服务端资源损坏或依赖不完整，仍会由 `dynamic_npc_quarantine` 隔离。

修改在角色下次进入该场景时生效，不会强制重建当前客户端已经生成的场景节点。

## 商品管理

- 独立“商品管理”Tab 从 `item.dsh`、`equip.dsh` 加载完整物品目录；
- 商城中的“秘宝道具”和“神兵利器”可按独立分区筛选、统计和管理；神兵利器分区包含武器、衣服、裤子、帽子、鞋子、束腰、披风和饰品。即使商品已下架，仍会保留在对应分区中，便于重新上架；
- 支持按装备/物品分类、名称或物品 ID 筛选，并按每页 50 项分页；
- 每个商品可以修改价格并设置上架或下架，保存后立即影响后续商城列表和购买结算；
- 下架商品同时从 `14/*` 商城分类页和 `17/1` NPC 商店列表中排除，直接伪造购买请求也不会成功；
- 价格覆盖用于服务端下发价格的 `14/*` 商城；`17/1` NPC 商店显示价格仍由客户端本地 DSH 决定；
- 下架不删除角色背包里的已有物品，也不影响后台赠送物品；
- 覆盖值保存在 MySQL `server_shop_items`。没有覆盖记录的商品继续采用 DSH 默认价格并默认上架；
- `806` 背包扩容的客户端显示/结算价格仍按容量档位固定为 20/40/60，页面对此给出提示。

## HTTP 行为与保护

- `GET /`：管理页面。
- `GET|POST /login`：登录页和密码验证。
- `POST /logout`：清除管理会话。
- `GET /healthz`：只读健康检查。
- `POST /action`：管理写操作；其他方法会返回 `405`。
- 页面无外部脚本或资源，响应禁止缓存并设置限制型 CSP。
- HTTP `Host` 只接受 `127.0.0.1` / `localhost`，如果浏览器携带 `Origin`，也必须
  来自同一本机管理端口，避免外部网页跨站提交加钱表单。
- HTTP 请求上限为 8 KiB，响应上限为 512 KiB。
- 本机管理连接的收发超时为 100 ms，避免残缺 HTTP 请求长期阻塞游戏端口。
- 管理监听失败不会导致 19090 游戏服务退出，启动日志会明确记录错误。

## 2026-07-18 运行验证

- `make -j2` 通过。
- 服务进程同时监听 `127.0.0.1:19090` 和 `127.0.0.1:19091`。
- `/healthz` 返回 `200` 和 JSON 健康状态。
- `/` 返回 `200`，包含账号、角色、加钱、加 W 币、创建账号及修改密码表单。
- `amount=0` 的无效加钱请求返回 `303` 中文错误提示，未修改现有角色余额。
- `GET /action` 返回 `405`。
