# 神仙壶登录数量与储量分离

## 现象

MySQL 中 `802 神仙壶`的 `item_count` 为正确的剩余 HP 储量 `50000`，角色登录后
打开背包却显示为 `18928`。角色 `10023` 的数据库复核还确认了三个独立实例，
序号分别为 `14/15/16`，每个实例的储量均为 `50000`；因此异常值不是持久化污染。

## 客户端证据

- `江湖OL.CBE:0x01039952` 解析 `30/21 iteminfo` 的每行
  `u32 itemId + i16 seq + u32 count + common-extra`，随后调用
  `0x0101918E`。
- `江湖OL.CBE:0x0101918E` 把上述 `count` 写入物品对象 `+242` 的 16 位数量槽，
  并把 `seq` 写入 `+276`。所以这里不能承载神仙壶的 32 位 `50000` 储量。
- `mmGameMstarWqvga.cbm:0x00000D04` 明确特判物品 802/803：完整的 `u32 count`
  分别写入对象 `+4/+8` 的 HP/MP 储量字段，同时把可见堆叠数量固定为 1。
- `江湖OL.CBE:0x01033544` 对 `7/11 info` 使用
  `row_count + i16 seq + u32 value`，按序号找到现有背包行，并把 802/803 的
  `value`写入同一个 `+4/+8` 储量字段。

## 修复合同

首次角色登录的背包引导现在按两个有序对象发送：

1. `30/21` 中 802/803 的网格 `count` 固定为 `1`，只建立正确的背包行和可见数量；
2. 紧随其后的 `7/11` 为每个壶按 `seq` 发送真实 32 位剩余储量。

普通物品继续在 `30/21` 中发送实际堆叠数。`17/1` 和通用 item-extra 中的壶类
运行时堆叠值也固定为 1。`7/11` 只随每次新的角色生命周期的一次性 `30/21`
发送，避免重复插入网格；部分使用过的壶同样可以恢复其实际剩余值。

## 验证

- MySQL 角色 `10023`：802 序号 `14/15/16` 均保持 `item_count=50000`。
- 登录回归校验 `30/21` 中三个 802 行均为 `wire_count=1`，并校验同一响应的
  `7/11` 中三个序号均为 `value=50000`。
- 重复 group 请求不重放 `30/21` 或储量种子；重新选择角色后会重新发送一次。
- `17/1` 中 802 的公共运行时数量字段为 1。
- `make -j2`通过并重新链接 `bin/main.exe`。
