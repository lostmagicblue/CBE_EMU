# 后台怪物管理

## 数据来源与边界

怪物 ID、默认等级、类型和任务掉落映射继续来自服务端内置怪物目录；场景中的
怪物名称和首次出现位置从真实 `automonster.dsh` 与 SCE2 战斗出生记录读取。
后台不修改客户端 SCE，也不允许创建客户端没有模板的任意怪物 ID，避免战斗包
引用不存在的 Actor 模板。

所有战斗路径原本都通过
`vm_net_mock_monster_stats_for_enemy()` 取得 HP、MP、攻击、防御、经验、铜钱和
掉落。怪物管理在这个统一入口增加 MySQL 覆盖，因此普通场景战斗、副本挑战、
挂机战斗和结算不需要各自维护一份属性。

## 数据表

`server_monsters` 以 `monster_id` 为主键，保存完整显式覆盖：

- 等级和怪物类型；
- HP、MP、攻击、防御；
- 经验和铜钱奖励；
- 掉落物品 ID 与百分比概率。

不存在覆盖行时继续使用服务端默认公式。“恢复服务端默认”会删除覆盖行，而不是
把一份默认值重复写入数据库。迁移脚本为
`server/mysql/migrate_add_monster_management.sql`；服务启动或首次读取怪物属性时
也会执行同等的 `CREATE TABLE IF NOT EXISTS`。

## 后台界面

后台新增“怪物管理”页：

- 左侧显示完整怪物目录，并可按 ID、名称或首次场景即时筛选；
- 名称和首次场景来自真实资源，只读；
- 右侧编辑等级、类型、战斗属性、结算奖励和掉落；
- MySQL 覆盖项用独立标记显示；
- 掉落物品必须存在于当前物品目录；无掉落时物品 ID 和概率必须同时为 `0`；
- HP、MP、攻击必须大于 `0`，掉落概率限制为 `0..100`。

- 标准 `make -j2` 已完成主功能编译、链接；最终安全上限调整时用户的模拟器
  客户端正在占用 `bin/main.exe`，因此最新源码另以
  `make -j2 TARGET=bin/main-monster-build.exe` 完整编译、链接并作为正式服务运行。
- 正式后台 `127.0.0.1:19091/admin-418yz6/?tab=monsters&monster=105`
  返回 200，页面包含“毒泥怪”、战斗属性、结算与掉落编辑区。
- 临时把怪物 105 的 HP 从 20 改为 21，页面运行时读值与 MySQL 行均立即变为
  21；随后执行“恢复服务端默认”，页面恢复 20，数据库覆盖行清零。
- 技能导师兼容回归通过；正式服务 PID `10284` 监听 `19090/19091`，stderr 为
  0 字节。
