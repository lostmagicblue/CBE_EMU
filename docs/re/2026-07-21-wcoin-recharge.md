# 用户中心 W 币充值

## 接口契约

实现依据 `http://pay.cbhub.top/api.html`：

- 创建订单：`POST /createOrder`
- 创建签名：`md5(payId + param + type + price + 通讯密钥)`
- 支付查询：`POST /checkOrder`，参数为云端 `orderId`
- 通知签名：`md5(payId + param + type + price + reallyPrice + 通讯密钥)`
- 异步通知成功响应：`success`

通讯密钥只从 `server_payment_config.secret_key`（或进程环境变量
`CBE_PAYMENT_KEY`）读取，不写入 HTML、日志或源码。

V免签后台需要分别配置微信 `wxpay` 与支付宝 `zfbpay` 收款码；支付宝可以
创建订单不代表微信通道已经配置。平台监控端离线时两个通道都会拒绝新订单。
创建失败时用户中心显示经过安全映射的平台原因，服务端只记录支付类型、HTTP
状态和原因代号，不记录通讯密钥、签名或完整请求表单。

## 用户流程

登录 `http://127.0.0.1:19091/` 后，用户可在“W币充值”卡片中选择角色、
整数元金额和支付方式。比例由数据库配置读取，当前为 `1 元 = 1000 W币`。

1. `POST /user/recharge/create` 先保存本地 `creating` 订单，再请求支付平台。
2. 创建成功后进入 `/user/recharge/order?id=...`，显示平台返回的
   `reallyPrice`、由 `payUrl` 在本站本地生成的二维码及支付链接，并每 3 秒刷新。
   页面明确要求实际付款金额与 `reallyPrice` 完全一致；金额多付或少付都无法
   自动关联本地订单并入账。页面同时按平台返回的 `timeOut` 显示逐秒倒计时；
   当前平台默认有效期为 5 分钟，归零后由查单流程同步标记为过期。
   微信二维码允许承载 `wxp://` / `weixin://` 内容，但只有 HTTP(S) 地址显示
   “打开支付链接”，自定义支付协议只用于本地二维码生成。
3. 订单页通过 `/checkOrder` 查询支付状态；支付平台也可直接请求
   `/payment/cbhub/notify`。
4. 查单返回的 `data` 含同一组签名回调参数，可在公网异步通知不可达时作为
   安全兜底。两条路径统一验签、核对本地订单金额/类型/随机参数并调用同一结算。
5. 入账成功页面 3 秒后自动返回用户中心。

如果支付平台后台未设置全局回调，可在 `server_payment_config.callback_base_url`
填写账号中心的公网地址；服务创建订单时会显式传递通知和返回地址。

## 入账和并发

`wcoin_recharge_orders` 使用以下状态：

- `0` 创建中
- `1` 等待支付
- `2` 已支付但等待入账
- `3` 已入账
- `4` 已过期
- `5` 创建失败

结算在一个 MySQL 事务中锁定订单和角色余额，更新角色 W 币后才把订单标记为
`credited=1`。重复通知返回成功但不会再次增加 W 币。待支付订单也会预留 W 币
数值空间，避免多个并行订单在 `UINT32` 余额上限处超额。

支付平台 DNS/HTTP 等待期间会暂时释放旧协议状态锁；本地延迟 2.2 秒的支付端
回归中，后台 `/healthz` 请求耗时约 1 毫秒，游戏请求不会被远端支付接口阻塞。
