# 后台任务管理与动态 NPC 任务绑定

日期：2026-07-20

## 目标

后台可以新增或编辑任务，并在“游戏内容管理”的动态 NPC 表单中直接选择该 NPC
提供的任务。客户端仍使用原生任务大厅、接取确认、活动任务列表和提交流程，不修改
客户端全局变量，也不改写 `task.dsh`。

## 数据模型

- `task.dsh` 仍是 133 条原版任务的安全基线。
- `server_tasks` 只保存后台编辑过的原版覆盖项和新增任务；同 ID 覆盖原版字段。
- `account_role_tasks` 继续只保存每个角色的状态与两组进度。
- `server_dynamic_npc_tasks` 保存动态 NPC 的可选任务 ID；解绑不会删除任务。

后台修改原版任务名称时，运行时仍保留 `task.dsh` 原名作为 XSE 标记别名。因此现有
XSE 中的“未接/已接/已完成”任务标记不会因为显示名称被编辑而失去映射。

## 客户端安全约束

延续已经确认的客户端结构：

- 活动任务记录由 `ParseItemDataFields(0x01046D24)` 解析，任务名、发布者、交付者
  的固定槽分别是 32、16、16 字节。
- 可接任务由 `DeserializeRoleInfo(0x01046E00)` 解析。
- NPC 任务选项使用 `action=4,value=task_id`，随后走 `6/10 -> 6/11`；提交仍走
  `6/4 -> 6/16`。
- 后台按 GBK 编码后的真实字节数校验名称和对话，不以 UTF-8 字符数代替。
- 条件类型只开放 `0=无、1=收集物品、2=击败怪物`，数量限制为 `u8`。
- 保留任务 ID `900001` 给历史协议回归，后台不能占用。

## NPC 状态机

绑定任务后，动态 NPC 不需要伪造 XSE：

1. 角色未接且满足等级/前置任务时，NPC 返回接取选项。
2. 接取后按后台“进行中对话”显示；战斗/掉落继续更新 MySQL 进度。
3. 两组条件达到数量后返回可提交选项，并附带 `6/6` 状态刷新。
4. 两组条件均为 0 时，下一次与交付 NPC 对话即转为可提交。
5. 提交奖励、放弃回收、任务列表和任务定位继续复用现有任务处理器。

动态任务参与场景 NPC 的任务优先级选择和 `6/14` 可接任务候选，因此客户端头顶
感叹号与任务大厅列表都使用同一份定义。数字前缀旧场景（如 `07泰山_01.sce`）
现在也会统一去掉可选 `.sce` 后缀再比较，避免后台 NPC 已保存却无法在对话时反查。

## 后台使用

1. 打开 `http://127.0.0.1:19091/`，进入“任务管理”。
2. 选择原版任务进行覆盖编辑，或点“新增任务”分配未占用 ID。
3. 填写名称、发布者、交付者、等级、条件、奖励和三阶段对话并保存。
4. 进入“游戏内容管理”，选择场景和动态 NPC，在“可接取任务”中选择任务并保存。
5. 角色重新进入场景后，NPC 会按角色任务状态显示感叹号、进行中或提交入口。

停用或删除仍有活动角色状态的任务会被拒绝；删除仍绑定动态 NPC 的自定义任务也会
被拒绝。原版覆盖项可以通过“恢复 task.dsh 默认”删除覆盖记录。
