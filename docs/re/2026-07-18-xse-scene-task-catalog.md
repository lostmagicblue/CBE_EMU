# 基于 XSE/SCE 的场景 NPC 与任务目录

日期：2026-07-18

## 目标

把场景 NPC 和任务从少量硬编码测试数据扩展为正式资源驱动：场景实体取自
`web/fs/JHOnlineData/*.sce`，对话与任务状态标记取自同目录的 `*.xse`，任务元数据
取自 `task.dsh`。服务运行时不读取 `tmp/all_xse_export` 或
`tmp/all_sce_export`。

## 资源清点

- 正式目录包含 149 个 XSE；`tools/inspect_xse.py` 全量检查结果为
  `149/149` 通过。
- 正式目录包含 196 个 SCE；195 个可由现有检查器完整解码。
- `b_01蓬莱仙岛.sce` 只有 29 字节，声明解压 16 字节但实际只有 15 字节，
  内容是一个不含 NPC 记录的 `SCE2` 占位资源；它不会生成交互 NPC。
- SCE 清点得到 148 个带 XSE 的 NPC 引用、140 个不同的 XSE 脚本。
- 129 个 XSE 含“未接”任务标记；去掉困难标记、空格以及半角/全角感叹号后，
  129 个名称都能唯一映射到 `task.dsh`。
- `task.dsh` 声明 138 行，其中 133 行同时具有非零任务 ID 和非空任务名，现均
  进入服务端任务目录。

## 已确认的二进制契约

IDA 实例按 `binary_name=江湖OL.CBE` 动态选择，没有硬编码实例 ID。

- `ParseItemDataFields(0x01046D24)` 消费活动任务记录，顺序为任务 ID、两个
  `u8`、任务名、给予者、等级、交付者、两组条件类型/ID/说明/进度/数量、
  一个 `i16` 和任务状态。客户端固定字符串槽分别为 32、16、16 字节。
- `DeserializeRoleInfo(0x01046E00)` 消费可接任务记录，顺序为任务 ID、难度、
  分类、任务名、场景 NPC 名、等级。
- `RegisterDisplayName(0x0100EEE0)` 只扫描四个 36 字节动态名称槽。四槽全满时
  它返回 `4`，而 `scene_parse_npcinfo_and_spawn_npcs(0x01037998)` 只排除 `-1`，
  会继续把第五码对应指针写到名称表之后。因此单次或重复累加的 `27/11` 都不能
  超过四名交互 NPC；直接提高数量会覆盖客户端内存。

## 实现

### 场景 NPC

- 场景进入和首次场景同步使用同一套 SCE 解析结果生成 `27/11 npcinfo`。
- 解析 SCE 中的 actor、XSE、显示名、方向和坐标；actor ID 由场景、脚本、
  名称和坐标稳定哈希生成。
- 铜雀台运行时键 `c00蓬莱仙岛_01` 对应的新 SCE 只包含传送门，不含 actor/XSE；
  同场景的旧资源 `00_蓬莱仙岛01.sce` 才保存大侠郭靖和郭芙蓉。NPC 扫描仅在
  精确 SCE 的交互目录为空时回退到这个可推导的旧格式别名，不会把两套目录叠加。
- 游戏内容核对确认铜雀台当前只应显示大侠郭靖。因此旧 SCE 回退还会应用
  `task0.xse + 大侠郭靖` 的精确目录筛选，郭芙蓉旧行不计入目录总数，也不下发。
- 旧铜雀台目录中的郭芙蓉引用 `n_girl.actor`。`bin/JHOnlineData` 是登录期间才会
  填充的可写客户端缓存，服务启动时可能为空，因此不能拿它作为正式目录的存在性
  判据；目录改为校验 `web/fs/JHOnlineData` 下载源。已确认不兼容的旧名仍显式映射
  为 `n_woman1.actor`，其他在服务端下载源中无法解析的模型直接跳过。
- 保留已经核实的服务端补充目录：蓬莱铸剑谷的欧冶子、小猴子，以及临安南宣门
  的王朝、马汉、胡斐。阶段性测试使用的“任务使者”不再加入任何场景目录。
- 服务端先保留最多 32 行完整场景目录，再根据角色 MySQL 任务状态选择客户端
  可安全实例化的四行：可提交/进行中任务的交付 NPC 优先，其次是当前可接任务
  的给予 NPC，最后才是普通脚本和闲聊 NPC；选中行仍按原 SCE 顺序下发。
- NPC 点击按完整目录中的稳定 actor ID 反查，不会因为接取任务后优先级改变而
  找不到已经生成在场景中的 NPC。
- 登录首场景和瞬移后场景都通过 scene-sync 的一次性 NPC seed 下发，避免只在
  跨地图后才出现。

### XSE 对话与任务状态

- 支持裸 `XSE0`、带四字节外层长度以及 type 1/type 2 LZSS 三种资源形式。
- 从 XSE 函数/命令尾表定位字符串池，提取普通对话和任务标记。真实对白必须匹配
  字节码中的直接调用序列：`1A 00 01 02 <stringIndex>` 后紧跟
  `1E 00 01 07 <SHOWDIALOG commandIndex>`，不会把任务选项标记误当对白。
- 对一个任务的普通脚本，已确认并支持 `接取/常态`、`接取/完成/常态`、
  `接取/续句/完成/常态` 三种布局；复杂多任务脚本暂时只采用可证明的最后一条
  常态对白，不猜测分支。
- 识别“未接 / 已接 / 已完成”任务分支，并结合角色等级、前置任务与 MySQL
  状态决定是否给出 `action=4` 的接取或提交选项。
- 纯对话任务只在交付者 XSE 的完成分支被点击时转为可提交，不用猜测坐标或
  伪造客户端状态。

### 任务协议与持久化

以下路径已改为使用 `task.dsh` 和 `account_role_tasks`：

- `6/14`：扫描当前场景已生成 NPC 的 XSE，返回当前可接任务。
- `6/1`：返回最多十条客户端可安全容纳的活动任务记录。
- `6/10`：任务目标与奖励说明。
- `6/11`：接取任务，并发放 DSH 的“给予物品”；已存在任务不能重复接取。
- `6/6`：任务完成状态更新。
- `6/4 -> 6/16`：提交任务，发放经验、金钱和奖励物品。
- `6/7`：放弃任务，同时回收接取时给予的任务物品。
- `6/12`：活动任务目的说明。

战斗首次结算时会把 DSH 条件同步到 MySQL：

- 条件类型 `1`：按本次成功进入背包的掉落物数量推进。
- 条件类型 `2`：按本次被击败的对应怪物数量推进。
- 两组条件都达到要求后把任务置为状态 `2`，下次任务列表/NPC 交互直接呈现
  可提交状态。
- 进度挂在已有的“每场战斗只结算一次”序列上，重复状态包不会重复累计。

  使用当前角色不满足等级/前置条件的任务，服务正确拒绝，未作为成功用例。

## 当前边界

- 这是结构解析器和任务状态机，不是完整的 XSE 虚拟机；复杂脚本中的非任务
  条件、跳转和副作用还未逐条解释。普通闲聊与三类任务标记已有确定行为。
- 当前确认真正超过四名有效坐标交互 NPC 的资源是 `c14蜀山_01.sce`：六名有效
  NPC（另有一条无坐标记录）。首次进场按任务状态挑选四名；若要在同一次场景
  生命周期中动态换入其余 NPC，仍需先逆出客户端名称槽和 scene node 的成对
  删除/释放契约，不能用重复 `27/11` 分页累加。
- 战斗结束不会在战斗包中插入未经证明的额外任务对象；任务进度在客户端下一次
  正常任务列表刷新或 NPC 交互时显示，以避免破坏战斗解析器。
