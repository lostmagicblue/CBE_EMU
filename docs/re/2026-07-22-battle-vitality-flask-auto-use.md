# 战斗结算自动使用神仙壶 / 逍遥壶

## 触发

角色携带 `802 神仙壶` 和/或 `803 逍遥壶`完成一场普通怪物战斗后，终局结算没有
自动补足角色缺失的 HP/MP，也没有下发壶的剩余储量。

## 已确认的客户端与资源合同

- `JHOnlineData/item.dsh`：802 的说明为可多次补充共 50000 点生命，803 同理补充
  50000 点法力；两者的“是否消耗”值为 `2`。
- `江湖OL.CBE:0x01033544`：802/803 不走普通物品减一；`1/7/11.info` 的
  `row_count + i16 seq + u32 remaining` 更新其内部储量，`remaining=0` 才删除背包行。
- 已有战斗手动物品使用也采用 `1/4/6 + 1/7/11`，而不是普通 `1/7/7 type=2`，
  因为后者会触发战斗客户端的第二次可见数量扣减。

## 首个偏差

普通战斗的终局响应由 `vm_net_mock_append_battle_status7_object` 生成 `1/4/7`，只处理
实验性环境变量恢复和奖励结算；它没有复用壶的储量恢复规划、没有更新背包储量，
也没有在同一终局响应追加 `1/7/11`。因此客户端和 MySQL 都不会观察到自动恢复。

## 修正计划

在每个 `4/7` 终局构造分支共用的封装中：

1. 仅对仍存活、通过正常 `4/7` 结算的角色，按 802 后 803 的背包行顺序，复用既有
   `min(缺失值, 储量, DSH 上限)` 规则，直到对应 HP/MP 补满或壶耗尽。
2. 让本场的战斗 HP/MP、角色持久状态和 `4/7.hp/mp` 恢复增量一致；死亡不会被壶
   隐式复活。
3. 紧随 `4/7` 追加一条仅含本次变更序号（包括归零行）的 `1/7/11`，不发送 `7/7
   type=2`，并将其计入 WT 对象数。
4. 继续由现有战斗结算保存一次角色与背包，避免新增独立的客户端状态修改或重复扣除。

## 验证目标

- 有缺失 HP/MP 时，终局 WT 包为 `4/7 + 7/11`（以及原有掉落刷新），`7/11` 的
  行值等于实际剩余储量。
- 满 HP/MP 时不减少储量，也不追加无意义的 `7/11`。
- 壶恰好耗尽时 `7/11` 显式下发该序号的 `0`，让客户端按已证实的分支删除背包行。
- `make -j2` 通过。

## 验证结果

- `make -j2`：通过。
- `tmp/battle-vitality-flask-auto-use-regression.php`：在独立服务（
  `CBE_BATTLE_ENEMY_COUNT=1`、`CBE_BATTLE_ENEMY_HP=1`）中通过。夹具角色以
  `20/120 HP`、`20/100 MP` 和分别剩余 `100/80` 的 802/803 进入战斗；终局日志
  记录 `mock_battle_auto_flask ... hp=100 mp=80 rows=2`，响应含 `4/7 + 7/11`，
  `7/11` 显式下发两个序号的零储量，MySQL 最终为 `120/120 HP`、`100/100 MP`，
  两个背包行均已删除。测试夹具已清理。
- 默认服务已重新启动于 `127.0.0.1:19090`，管理健康检查
  `http://127.0.0.1:19091/healthz` 返回 HTTP 200。
