# 动态 NPC 商店、修理与技能导师

## 客户端证据

- `task_hall_activate_selected_entry/0x010492B0` 按对话选项中的 `action`
  分派。`action=1` 会发送 `26/1 {type=2,id=value}`，因此可安全用于同一个
  `26/1` 对话界面中的服务子菜单，不需要修改客户端状态或全局变量。
- `action=3` 是原版普通商店入口：发送 `26/3 {value}`；响应由
  `ParseShopItemData/0x01038976` 读取。购买再发送 `26/2 {type,value}`，响应
  由 `ParseItemRewardData` 读取。该链路与酷币商城 `14/*` 不同，不能混用。
- `sub_101CA42/0x0101CA42` 构造装备修理请求 `7/29 {type}`；单件修理
  (`type=2`) 还会携带 `seq` 和 `id`。`HandleRepairResponse/0x01028C00`
  读取 `type`、`repairnum`、`coolmoney`。
- 已学技能列表由 `12/1 {learnednum,learnedskill}` 刷新；`learnedskill` 是
  连续的 `u32 skill_id` 序列。原服务此前按角色等级临时推导列表，没有持久化
  学习状态。
- 旧 SCE 中同一种 `n_shop.actor` 商店入口通过 field `0x20` 的商店编号区分
  目录。生产场景可见 `13/17/21/25=武器商人`、`18/22/26=铠甲商人`、
  `19/23/27=皮甲商人`、`20/24/28=布甲商人`，临安的
  `33=药水/药物商人`。因此“武器商人”不是全装备商人；腰带属于防具目录，
  药品也有独立商店。测试地图中还保留过旧编号 `1=药水商人`，说明商店编号
  同时承载地区/等级目录，不能直接当成稳定的动态 NPC `npc_kind` 枚举。

## 当前服务契约

`server_dynamic_npcs.npc_kind` 只接受以下值：

- `0`：普通／任务 NPC；
- `1`：武器商人；
- `2`：装备修理；
- `3`：技能导师；
- `4`：防具商人（含腰带）；
- `5`：药品商人。

非零类型在 NPC 原有任务选项之外增加一个 `action=1` 服务入口。服务子菜单仍
返回合法的 `26/1` 对话对象，并使用 `0xE1` 到 `0xE9` 高字节的私有 value
命名空间。所有低 24 位商品或技能 ID 在状态变更前重新对照 DSH 目录校验。

武器商人先显示剑、匕首、法杖三个子类，避免目录按 ID 排序后前七行全部是剑。
防具商人显示头盔、衣甲、披风、腰带、护腿、鞋靴和戒指七个子类。药品商人
只列出 `item.dsh` 类别 `10` 的药品与消耗品。每个商品子类每页最多五件，并
在七选项的客户端安全上限内附加上一页/下一页。价格沿用 `server_shop_items`
覆盖或 DSH 默认值；购买成功扣除角色铜钱、写入关系型背包，并在同一响应中
追加 `7/7 {type=1,iteminfo=<本次获得的一条增量行>}`。这是场景 NPC
对话回调实际消费的物品管理器更新路径；`17/1` 只保留给背包／商店列表模块
主动请求的全量列表，不能附带在 `26/1` 对话响应后充当刷新。

装备耐久保存在 account_role_equipment_durability。当前 mock 的玩法策略是在
每个已结束战斗对已装备槽扣 1 点；换装会按新物品重置该槽为 100/100。修理费用
为缺失耐久点数，修理成功后扣铜钱并恢复到最大值。内置欧冶子默认配置为装备修理
NPC。该扣减频率和上限尚未由原服包证明，详见
2026-07-22-battle-equipment-durability.md 的“固件复核”。

技能保存在 `account_role_skills`。没有关系行的角色首次加载时只初始化一项本职业
1 级基础技能；此后无论角色升到多少级都不会自动增加技能，后续技能只能由技能
导师学习。导师只显示当前职业、等级满足且尚未学习的技能。`skill.dsh` 的 `价值` 列作为导师学习
费用，学习时同时校验并扣除角色铜钱；成功后在同一响应追加 `12/1` 刷新。若当前
等级没有新技能，导师会显示当前等级以及下一技能的名称、要求等级和铜钱费用，
不再只返回“当前没有可以学习的技能”。

### 1 级幻剑无技能选项的核对

`LoadSkillDataSheet/0x0103550E` 按角色职业读取 `skill.dsh`，并明确装载 `ID`、
`名称`、`说明`、`等级`、`价值`、`使用限制` 等列；角色职业到 DSH 职业仍使用
`raw_job = role_job - 1`。本次运行角色 `guest00023/10027` 为职业 `2`（幻剑）、
等级 `1`，关系表只含初始技能 `101 风舞刃行1`。幻剑下一条尚未学习的技能为
`111 混沌鬼气1`，要求等级 `3`，价值 `160` 铜钱。因此旧提示中的
“无可学习技能”并非职业映射失败，而是角色尚未达到 3 级。

修正后的诊断日志会同时输出 `job`、`level`、`skill_eligible`、`skill_learned`、
`skill_level_locked`、`next_skill`、`next_level` 和 `next_price`。对应回归日志为：

```text
mock_npc_service action=skill-list ... role=10027 job=2 level=1 ...
skill_eligible=0 skill_learned=1 skill_level_locked=21
next_skill=111 next_level=3 next_price=160 ...
```

## 后台与数据库

Web 动态 NPC 管理页把原来的任意数字“类型”改为六项下拉选择，保存接口也限制
范围为 0..5。完整新库使用 `server/mysql/schema.sql`；已有数据库执行
`server/mysql/migrate_add_npc_services.sql`。运行时仍执行同等的
`CREATE TABLE IF NOT EXISTS`，但迁移脚本保留完整外键约束。
