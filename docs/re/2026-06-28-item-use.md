# 2026-06-28 Item Use

## Problem

Using the HP medicine shown by the client as `小长春散` / local `item.dsh`
row `304 小长命散` crashed because the mock server did not handle the
in-game item-use request:

```text
WT 7/1 len=42 objects=1 first=1/7/1:33
```

The assert happened before the callback could parse any response:

```text
unhandled wt=7/1 ... first=1/7/1:33
```

## Request Evidence

- Runtime signature: one WT object, `major=1 kind=7 subtype=1`, payload length
  `33`.
- `mmGameMstarWqvga.cbm` local strings around the request builder include
  `type`, `id`, and `num`. The 33-byte payload size matches three typed fields:
  `type` as u8, `id` as u32, and `num` as u32.
- Because `id` can be either an item id or the target role id, the server parser
  treats `itemId/itemID/itemid` as explicit item selectors, accepts `id` only
  when it matches a known usable item or active backpack row, and otherwise
  uses `type+num` as an effect request.

Current effect mapping for the fallback `type+num` shape:

```text
type=1 or unknown -> HP change
type=2            -> MP change
type=3            -> EXP change
```

Usage count is not read from `num`; `num` is the effect value in this packet.
The consume count stays `1` unless an explicit `count/usecount/useCount` field
appears.

## Client Parser Evidence

`mmGameMstarWqvga.cbm:0x11CE` is the normal game network callback used by this
screen. For data event `7` it handles:

- `kind=7 subtype=7` by calling `sub_D04`;
- `kind=16 subtype=1/2/3` for scene/confirm/hint branches;
- `kind=14 subtype=3` for shop purchase completion.

It does not handle a response object `7/1`, so the mock response must not echo
the request subtype.

`mmGameMstarWqvga.cbm:0x0D04` parses `7/7`:

```text
field type       u8
field iteminfo   raw stream

iteminfo:
  u8 row_count
  repeat:
    i16 seq
    u32 itemId
    u32 count
    common item-extra block
```

`type=1` calls the add/update item-manager path and has negative history as an
initial backpack refresh because it duplicated an existing stack. For item use
the mock sends `type=2`, the alternate item-manager path, after updating the
server-side role DB.

## Server Behavior

Implementation in `src/mock-server.c`:

- Load item effects from `JHOnlineData/item.dsh` or `bin/JHOnlineData/item.dsh`.
- Read columns:
  - `5 类别`
  - `6 等级`
  - `10 堆叠数`
  - `15 生命变化`
  - `16 法力变化`
  - `17 经验变化`
- Treat category `10` or any nonzero HP/MP/EXP effect as usable.
- If `item.dsh` cannot be loaded, fallback rows cover common HP/MP medicines:
  `301..305` and `321..325`.
- Match the item by explicit `itemId/seq`, by usable `id`, or by matching the
  requested HP/MP/EXP effect against the active role backpack.
- Apply HP/MP/EXP to the active role, cap HP/MP to derived max values, normalize
  level after EXP changes, consume one item from the active role backpack, and
  save `nvram/jhol_mock_roles.bin`.
- Apply the effect only after the active role backpack item is consumed. A stale
  client request for an already-empty stack must clear the visible stack but
  must not heal or grant EXP again.
- Return:
  - `1/7/1 { result=1, type=<request-type>, id=<itemId> }`
  - `1/7/7 { type=2, iteminfo=<seq,itemId,remaining,extra> }`
  - followed by `1/7/11 { info=<row_count,seq,new_count> }`
- If no item row can be identified, return a safe `1/16/2 result=4` hint
  instead of crashing or applying the item effect.

## Quantity Refresh Follow-Up

Manual runtime after the first implementation showed that the HP recovery path
worked, but the visible backpack stack did not decrement:

```text
mock_item_use item=304 seq=2 count=1 remaining=1 ... applied=1 consumed=1 refresh=17/1
mock_item_use item=304 seq=2 count=1 remaining=0 ... applied=1 consumed=1 refresh=17/1
mock_item_use item=304 seq=2 count=1 remaining=0 ... applied=1 consumed=0 refresh=17/1
```

This proves two things:

- The role DB was decremented correctly on the first two uses.
- The client-visible backpack row was stale, because it still sent a third use
  request for the same `seq=2` after the server stack reached zero.

The failed response path is consistent with the parser split:

- `mmGame:0x0D04` accepts `7/7`, but `type=2` calls an alternate item-manager
  method without passing the parsed row to it.
- `mmGame:0x11CE` ignores kind `17`, so an appended `17/1` full-list refresh is
  not consumed while this callback owns the response.

`JianghuOL.CBE:0x01033544 HandleItemOperationResponse` is reached by the main
CBE kind-7 dispatch before the mmGame callback. For subtype `11` or `12` it
reads field `info` as a raw stream:

```text
u8 row_count
repeat row_count:
  i16 seq
  u32 new_count
```

It then looks up the active backpack row by `seq` and writes the new count into
the item row.

Follow-up runtime showed that `7/11` fixes the visible stack count, but if the
new count is zero it does not call the ordinary item-manager delete/decrement
path for normal medicines. The bottom statistic still showed `2/40` because the
row remained allocated in the client item list.

The same CBE handler has a subtype `1` success path:

```text
7/1:
  result
  type
  id
```

When `R9+0x9494` holds the pending item-use row, `result=1` clears the pending
state and calls the item-manager operation at `itemCtrl+56` with
`type,id,count=1`. That is the client-owned path that decrements/removes the
row and updates the occupied-slot counter. The response now sends:

```text
7/1 result=1,type,id
7/7 type=2,iteminfo
7/11 info
```

`7/1` lets the client update list occupancy, while `7/11` keeps the final
visible stack count aligned with the server-side role DB.
