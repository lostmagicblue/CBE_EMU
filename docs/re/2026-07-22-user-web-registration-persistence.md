# 用户前台注册失败：账户全表重写

## 根因

注册校验通过后，用户前台会调用账户持久化函数。
该函数在事务中执行 DELETE FROM accounts 后再回写全部内存账户。

wcoin_recharge_orders.account_id 通过 fk_wcoin_recharge_account 外键引用
accounts，且已有充值订单。因此 MySQL 以 1451 拒绝删除父账户行，事务回滚，
前台将该保存失败显示为“账号注册失败，请稍后重试”。

## 修复契约

没有账户删除业务入口；注册、改密和迁移只需新增或更新账户。保存必须在原事务内
逐条 upsert，而不是删除并重建父表。任一写入失败仍整体回滚，不能伪造注册成功。

## 验证

验证新账号注册、重复注册、改密，以及已有充值订单仍可读取。

## 范围

本轮保持既有密码存储格式和前台文案契约，仅修复关系数据库写入范围。

## 验证结果

已在本地服务 `127.0.0.1:19091` 完成真实 HTTP 验证。新账号注册返回 303，
重复注册返回 409 且页面包含“该账号名已被注册”；同一会话改密返回 303，旧密码
登录返回 401，新密码登录返回 303。测试期间账户数由 24 临时增至 25 后已清理回
24；充值订单数始终为 7。服务随后重启，使内存账户缓存与数据库一致。

## 可扩展性修正

上一版以逐条 upsert 替代了全表删除，恢复了外键完整性，但注册和改密仍会遍历
整个内存账户数组。该遍历属于旧的快照落盘模型，不是关系数据库正常写入所需的
契约；账号数增长时，单次账号操作会线性放大为全表 SQL 写入。

正常路径应只写本次变更的账户：注册使用单行 INSERT，改密使用带 account_id
条件的单行 UPDATE，并在数据库成功后才更新内存镜像。只有首次旧文件迁移仍可
使用事务内的全量 upsert，因为它本来就是一次性快照导入。验证需要确认正常注册
或改密只发出一条账户 DML，且原有注册、改密和充值订单完整性回归保持通过。

实施后，正常路径不再调用全量保存函数：注册写入成功后才追加内存记录，改密写入
成功后才替换内存密码。全量函数只被旧文件 `legacy-migrate` 调用。

本地 HTTP 回归中，新账号注册和改密均返回 303；重复注册返回 409 并显示账号已
存在；旧密码登录返回 401，新密码登录返回 303。测试账户清理后账户数恢复为 24，
充值订单数仍为 7，服务已重启并通过 healthz 检查。
