# CBE_EMU 项目协作规范

## 故障处理必须根因优先

本规范适用于客户端闪退、断言、解包错误、界面卡住、状态错乱，以及由模拟服务端响应、事件投递、资源更新或持久化逻辑引起的其他异常。

### 修改代码前的准入条件

处理故障时，必须先理顺完整业务链路：

1. 固定可重复的触发步骤，保留原始客户端日志、服务端日志、请求和响应字节、崩溃寄存器或调用位置。
2. 串起客户端请求、服务端 detector/handler/builder、响应对象与字段顺序、网络事件投递、客户端 parser/callback 以及相关状态生命周期。
3. 明确预期行为、实际行为和二者第一次发生偏离的位置。崩溃 PC 通常只是最终症状，不能自动视为根因。
4. 使用运行时证据、真实包结构、IDA 反编译/汇编和资源数据交叉验证字段语义与时序；同时记录已排除的假设。
5. 在调查文档中写出可检验的根因陈述：触发条件、被违反的契约、首个错误状态、证据和计划修改点。

如果还无法确定第一次偏离发生在哪里，只允许增加窄范围、可删除的取证日志或更新调查文档，不得先改变业务行为来“试着不崩”。

### 修复要求

- 修复最早被违反的协议、数据来源、对象生命周期、状态归属或事件顺序；修改应位于真正拥有该契约的层级。
- 不得以“客户端不再闪退”作为唯一完成标准。客户端必须走过预期 parser 分支并产生正确的后续 UI、状态或请求。
- 不得盲目丢弃请求或响应对象、吞掉错误、返回伪造成功、硬编码默认场景/坐标/角色/资源、重复投递、自动重试、强制重置状态，或按崩溃地址添加特判。
- 不得通过修改客户端内存、全局变量、寄存器、PC/LR 或 CBE/CBM 代码绕过真实流程。
- 权威数据缺失或语义未确认时，应显式记录 `unresolved` 并继续取证，不能用看似可用的默认值掩盖缺口。

抑制或兜底逻辑只有同时满足以下条件才能使用：

1. 有客户端 parser、真实协议包、资源格式或既有服务端行为证明它属于正确契约或明确的兼容边界。
2. 作用范围足够窄，不会改变正常路径，也不会隐藏新的未知请求或数据错误。
3. 有可观察日志、调查文档和针对性回归测试，并说明为何根因修复后仍然需要它。

### 验证与交付

- 复测原始失败路径，并覆盖相邻路径、重复进入/退出、重新登录和多客户端状态隔离等与问题相关的场景。
- 检查没有引入加载停滞、重复回调、重复动画、状态泄漏、错误默认值或新的高频延迟。
- 修改代码后执行 `make -j2`；文档修改应检查链接、术语和规范间没有冲突。
- 将触发条件、证据、根因、修改点、验证结果和仍未知的风险记录到 `docs/re/`，便于后续继续取证，而不是只在代码里留下特判。
